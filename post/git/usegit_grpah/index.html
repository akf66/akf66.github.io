前言

相比命令行操作，使用图形化 Git 工具进行 cherry-pick 更加直观和便捷。本文将通过实际截图演示如何使用图形化工具（如 GitKraken、SourceTree、VS Code Git Graph 等）完成 cherry-pick 操作。

使用场景

在实际开发中，我们经常遇到这样的情况：

  • 在测试分支开发了新功能
  • 测试通过后，只想将这个功能的提交应用到生产分支
  • 不想合并整个测试分支（可能包含其他未完成的功能）

这时候 cherry-pick 就是最佳选择！

图形化操作步骤

步骤 1：在测试分支添加新功能

首先，在测试分支（test/dev）上进行开发：

  • 完成新功能的代码编写
  • 提交更改（commit）
  • 在测试环境中验证功能正常

关键点：

  • 确保提交信息清晰明了，方便后续识别
  • 每个功能尽量独立提交，避免一个提交包含多个不相关的更改
  • 充分测试后再进行 cherry-pick

步骤 2：选择要 Cherry-Pick 的提交

在图形化工具中：

  1. 找到你想要应用的提交记录
  2. 右键点击该提交
  3. 选择 “Cherry Pick Commit” 选项

⚠️ 重要提醒：

  • 不要直接点击 “Push to main”！
  • 此时只是选择提交，还没有应用到目标分支
  • 需要先切换到目标分支再执行 cherry-pick

步骤 3：切换到目标分支并应用提交

接下来的操作：

  1. 切换到目标分支（如 release、main 或其他生产分支）
  2. 执行 cherry-pick 操作，将选中的提交应用到当前分支
  3. 图形化工具会自动创建一个新的提交

Cherry-Pick 的优势：

  • ✅ 只接入特定的更改，产生一个新的提交
  • ✅ 不会合并整个分支的所有提交
  • ✅ 保持目标分支的提交历史清晰
  • ✅ 避免引入未完成或未测试的代码

与 Merge 的区别：

  • Merge（合并分支）：会将源分支的所有提交都合并过来
  • Cherry-Pick：只选择特定的提交应用到目标分支

步骤 4：查看最终效果

Cherry-pick 完成后，你可以看到：

  • 目标分支上出现了一个新的提交
  • 这个提交包含了你选择的更改
  • 提交的 hash 值与原提交不同（这是正常的）
  • 分支历史保持清晰，没有不必要的合并记录

验证步骤：

  1. 检查提交历史，确认新提交已经出现
  2. 查看代码更改，确保内容正确
  3. 在目标分支上重新测试功能
  4. 确认无误后，推送到远程仓库

处理冲突

如果 cherry-pick 过程中出现冲突，图形化工具通常会：

  1. 高亮显示冲突文件
  2. 提供冲突解决界面
    • 可以选择保留哪一方的更改
    • 或者手动编辑合并
  3. 解决冲突后继续
    • 标记冲突已解决
    • 点击 “Continue Cherry-pick”
  4. 或者放弃操作
    • 点击 “Abort Cherry-pick”
    • 恢复到操作前的状态
